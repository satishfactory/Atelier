## Feature 1 — Deployment
**Effort: 2–3 hours · Blocks: everything below**

### What
- Frontend (React/Vite) → Vercel
- Backend (Express server.js) → Railway
- Environment variables configured in both platforms

### Steps

**Step 1A — Vercel (frontend) — 30 min**
```bash
npm install -g vercel
vercel login
cd ~/Projects/atelier
vercel
```
- Set root directory to project root
- Build command: `npm run build`
- Output directory: `dist`
- Add env vars in Vercel dashboard:
  - VITE_SUPABASE_URL
  - VITE_SUPABASE_ANON_KEY

**Step 1B — Railway (Express server) — 45 min**
- railway.app → New Project → Deploy from GitHub
- Root directory: project root
- Start command: `node server.js`
- Add env vars:
  - SUPABASE_URL
  - SUPABASE_SERVICE_KEY
  - ANTHROPIC_API_KEY
  - OPENAI_API_KEY (for Whisper)
  - PORT (Railway sets this automatically)

**Step 1C — Wire frontend to deployed backend — 15 min**
- In `src/lib/api.js`: change base URL from `localhost:3001` to Railway URL
- Use env var: `VITE_API_URL=https://your-app.railway.app`
- Add to Vercel env vars

**Step 1D — Test checklist**
```
[ ] Home screen loads with morning briefing
[ ] Gallery loads images from Supabase Storage
[ ] Evaluate tab streams a response
[ ] Blog generates and saves
[ ] Upload photo works
```

### Files to touch
- `src/lib/api.js` — swap base URL to env var (one line change)
- `.env.production` — add VITE_API_URL
- `server.js` — confirm PORT is read from `process.env.PORT || 3001`

## Feature 5 — Companion memory / continuity
**Effort: 2–3 hours · Prerequisite: rolling_summary injection (Phase 4 audio/video)**

### What
The companion references insights across paintings, not just within one. "You've been struggling with backgrounds across three paintings — here's what I notice." This is done by injecting a cross-painting summary into the evaluate_painting context.

### Approach
No new table needed. Uses existing data. Adds a new field to `artist_profiles`: `cross_painting_summary TEXT`.

Updated after every evaluation: Claude reads the last 5 painting rolling_summaries and writes a synthesised cross-painting observation paragraph.

### New callType: `update_artist_summary`
Add to `/api/evaluate` callType switch:
```javascript
case 'update_artist_summary':
  // Input: last 5 paintings' rolling_summaries
  // Output: 200-word cross_painting_summary
  // Write to: artist_profiles.cross_painting_summary
  // max_tokens: 300
```

### Context injection
In `evaluate_painting` context assembly, add after rolling_summary:
```javascript
const artistProfile = await getArtistProfile(artistSlug);
// inject: artistProfile.cross_painting_summary
```

### Steps
1. Add `cross_painting_summary TEXT` to artist_profiles (5 min)
2. Add `getArtistProfile()` and `updateArtistSummary()` to supabase.js (15 min)
3. Add `update_artist_summary` callType to server.js (20 min)
4. Trigger `update_artist_summary` async after each `evaluate_painting` call (10 min)
5. Inject `cross_painting_summary` into evaluate_painting context (10 min)
6. Test: evaluate two paintings, check that third evaluation references the others (20 min)

## Feature 7 — Authentication / multi-user
**Effort: 4–6 hours · Do last**

### What
Email/password login via Supabase Auth. Each artist sees only their own data. RLS policies enforce isolation.

### Steps

**Step 7A — Enable Supabase Auth (15 min)**
- Supabase dashboard → Authentication → Enable Email provider
- No magic links for now — password only

**Step 7B — Add login screen (45 min)**
- `src/components/LoginScreen.jsx` (max 100 lines)
- Email + password fields, sign in button
- On success: store session, redirect to HomeScreen
- On fail: show error message

**Step 7C — Auth context (30 min)**
- `src/lib/auth.js` — `signIn()`, `signOut()`, `getSession()`
- Wrap App.jsx with auth check: if no session → LoginScreen, else → app

**Step 7D — RLS policies (60 min)**
For each table that has artist_slug or user data:
```sql
-- Example for paintings:
ALTER TABLE paintings ENABLE ROW LEVEL SECURITY;
CREATE POLICY "artist_isolation" ON paintings
  FOR ALL USING (artist_slug = auth.uid()::text);
```
Apply same pattern to: painting_sessions, painting_images, companion_conversations, blog_posts, studio_states, score_history, session_frames.

**Step 7E — Update all queries (60 min)**
Supabase client auto-attaches auth header when user is logged in. RLS handles the rest. No query changes needed IF artist_slug matches auth.uid(). May need to update insert calls to set artist_slug = auth.uid().

**Step 7F — Test**
Create two test accounts. Confirm each sees only their own paintings.
